## 一、产品定位

### 1.1 一句话定位

**Web3 领域的个性化信息聚合订阅服务**

接管用户的信息源，用 AI 筛选出对每个用户最有价值的信息，每天一份简报，10 分钟看完。我们认为对任何用户的信息需求，对该最有价值的主要信息几乎可以在10分钟内快速浏览完，花时间的是从一大堆噪音中找到这加起来10分钟的信息内容。

### 1.2 核心价值洞察

> **关键洞察**：用户每天花几个小时刷各种信息源，但对于每个用户来说，当天**真正有价值、非看不可的信息**，其实就是 10 分钟能看完的量。绝大多数信息对他都是**噪音**。
>
> **我们的价值**：帮用户从海量噪音中**精准提取**出那些对他真正有价值的信息，让他**既节省时间，又不错过重要内容**。

| 用户痛点 | 核心洞察 | 我们的价值 |
|----------|----------|------------|
| 每天花 3+ 小时刷信息 | 真正有价值的内容其实只有 10 分钟量 | 帮你**找到**那 10 分钟的精华 |
| 怕错过重要信息 | 重要信息被噪音淹没了 | **精准提取**，确保不遗漏,无语义重复 |
| 90% 是噪音 | 用户自己很难分辨价值 | AI 帮你**识别和过滤** |
| 信息分散多平台 | 到处找太累 | **一站式聚合**，统一呈现 |

### 1.3 价值可感知设计

**核心原则**：让用户在每份简报中**清晰感知到**我们创造的价值。

每份简报必须包含的「价值展示」模块：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📊 今日为您做了什么

• 监控信息源: 58 个
• 扫描原始信息: 847 条
• 为您精选: 15 条（仅占 1.8%）
• 今日为您节省: 约 2.5 小时
• 累计为您节省: 47 小时

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**设计意图**：
- 让用户知道我们**真的在帮他筛选**，而不是随便推几条
- 用数据量化价值（节省 X 小时）
- 累计数据增强用户粘性和付费意愿

### 1.4 产品形态

| 项目 | 说明 |
|------|------|
| **服务类型** | SaaS 订阅服务（我们运营，用户注册即用） |
| **用户操作** | 注册 → 简单设置偏好 → 被动接收简报 |
| **交付形式** | 每日信息简报（Telegram 推送） |
| **商业模式** | 免费 + 付费增值（$20/月，支持 USDC） |

## 二、核心设计原则

### 2.1 产品原则

| 原则 | 说明 |
|------|------|
| **我们是聚合器** | 用户关心什么，我们就去拿什么，不重复造轮子 |
| **More intelligence, less structure** | 让 AI 决定怎么完成任务，减少人类预设的流程限制 |
| **渐进式验证** | 预判 → 嵌入选项 → 用户反馈 → 决定是否加强 |
| **给用户做选择题** | 用户不知道要什么，我们给预判选项，让他们选 |

### 2.2 "More intelligence, less structure" 落地

**核心思想**：充分相信 AI 能力，让产品能力随 AI 进化而增长，不被人类预设的规则限制。

| 环节 | 传统做法 | 我们的做法（AI 主导） |
|------|----------|----------------------|
| **理解用户偏好** | 用户填表选标签 | AI 理解用户意图 + 行为学习 |
| **判断信息价值** | 关键词匹配 | AI 语义理解，综合判断 |
| **内容筛选** | 固定规则 | AI 判断：质量 × 相关度 × 时效性 |
| **简报生成** | 固定模板 | AI 决定怎么组织、突出什么 |
| **持续进化** | 人工调整规则 | **用户反馈 → AI 自动学习** |

**架构原则**：
1. **最小硬编码** — 只在必要处写死逻辑，其他交给 AI
2. **AI 能力可插拔** — 模型可升级，产品能力随之增长
3. **Prompt 即产品** — 核心逻辑通过 Prompt 定义，可快速迭代
4. **反馈闭环** — 用户反馈 → AI 学习 → 产品进化

### 2.3 用户反馈学习闭环（核心机制）

> **核心理念**：让用户成为产品进化的一部分。每一次用户反馈，都让 AI 更懂这个用户。

#### 学习闭环设计

```
┌─────────────────────────────────────────────────────────────────┐
│                    用户反馈学习闭环                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   用户收到简报                                                   │
│        ↓                                                        │
│   用户给反馈（低负担方式）                                        │
│   • 整体：👍有用 / 👎不太行                                      │
│   • 具体：某条信息"不感兴趣"/"很有用"                            │
│   • 补充：快速选择不满意原因                                     │
│        ↓                                                        │
│   系统记录反馈                                                   │
│        ↓                                                        │
│   AI 分析反馈，更新用户偏好画像                                  │
│        ↓                                                        │
│   下次生成简报时，AI 使用更新后的用户画像                         │
│        ↓                                                        │
│   简报更符合用户喜好 → 用户更满意                                │
│        ↓                                                        │
│   （循环）                                                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 反馈收集方式（低负担设计）

**原则**：用户 5 秒内完成反馈，不增加负担

**方式 1：整体评价**
```
今天的简报对你有帮助吗？
[👍 有用]  [👎 不太行]
```

**方式 2：不满意时的快速原因选择**
```
（用户点击"不太行"后弹出）

哪里不够好？（可多选）
□ 推的内容我不感兴趣
□ 漏掉了重要信息
□ 信息太多/太杂
□ 信息太少/不够
□ 其他：[简短输入]
```

**方式 3：对具体信息的反馈（可选）**
```
每条信息旁边可以点击：
[👎 不感兴趣]  [⭐ 很有用]
```

#### AI 如何利用反馈

| 反馈类型 | AI 学习内容 | 下次调整 |
|----------|-------------|----------|
| 用户点"不感兴趣"某条信息 | 记录该类信息/来源不受欢迎 | 降低该类信息的推送权重 |
| 用户点"很有用"某条信息 | 记录该类信息/来源受欢迎 | 提高该类信息的推送权重 |
| 用户说"漏掉重要信息" | 筛选阈值可能太高 | 适当放宽筛选，多推一些 |
| 用户说"信息太多太杂" | 筛选阈值可能太低 | 收紧筛选，只推最重要的 |
| 用户说"不感兴趣" | 某领域/来源不匹配 | 调整用户画像中的领域权重 |

#### 用户画像动态更新

每个用户有一个**动态画像**，包含：

```yaml
user_profile:
  # 初始设置（用户注册时选择）
  initial_interests: ["DeFi", "以太坊", "融资动态"]
  initial_projects: ["ETH", "ARB", "OP"]
  
  # AI 学习到的偏好（基于反馈持续更新）
  learned_preferences:
    liked_sources: ["@VitalikButerin", "@lookonchain"]  # 用户标记"有用"的来源
    disliked_sources: ["@xxx"]                          # 用户标记"不感兴趣"的来源
    liked_topics: ["L2 技术进展", "大额融资"]           # 用户喜欢的主题
    disliked_topics: ["Meme 币", "价格预测"]            # 用户不喜欢的主题
    
  # 筛选偏好（AI 学习调整）
  filter_preferences:
    strictness: "medium"      # loose / medium / strict
    preferred_count: 15       # 用户偏好的信息条数
    
  # 反馈历史摘要（AI 总结）
  feedback_summary: "该用户更关注技术进展和融资动态，对价格分析和 Meme 币不感兴趣。偏好精简的信息，不喜欢太多条目。"
```

#### Prompt 动态注入

每次生成简报时，AI 的 Prompt 会自动包含该用户的学习画像：

```
你正在为以下用户生成今日简报：

## 用户画像
- 关注领域：DeFi, 以太坊, 融资动态
- 关注项目：ETH, ARB, OP

## AI 学习到的偏好（重要！）
- 该用户喜欢的来源：@VitalikButerin, @lookonchain
- 该用户不喜欢的来源：@xxx
- 该用户喜欢的主题：L2 技术进展, 大额融资
- 该用户不喜欢的主题：Meme 币, 价格预测
- 用户偏好：精简信息，不喜欢太多条目

## 历史反馈总结
该用户更关注技术进展和融资动态，对价格分析和 Meme 币不感兴趣。

请基于以上用户画像，筛选和组织今日简报...
```

#### 进化效果

| 使用阶段 | AI 对用户的理解 | 简报质量 |
|----------|-----------------|----------|
| 第 1 天 | 仅知道用户选择的领域 | 基础匹配 |
| 第 7 天 | 学习到用户喜欢/不喜欢的来源 | 更精准 |
| 第 30 天 | 深度理解用户偏好模式 | 高度个性化 |
| 持续使用 | 持续进化，越来越懂用户 | 越用越好 |

### 2.4 反馈学习闭环的技术实现（More Intelligence 方式）

> **核心理念**：不写规则让代码执行，而是让 AI 分析反馈、AI 总结偏好、AI 决定如何使用。

#### 传统方式 vs More Intelligence 方式

| 环节 | 传统做法（Structure） | More Intelligence 做法 |
|------|----------------------|------------------------|
| **分析反馈** | 人写规则：if 不喜欢X → weight -= 0.1 | **让 AI 分析**反馈意味着什么 |
| **表达偏好** | 结构化标签 + 数字权重 | **AI 用自然语言**描述理解 |
| **应用偏好** | 代码硬编码过滤逻辑 | **AI 自己决定**如何使用偏好 |
| **进化能力** | 需要人工迭代规则 | 换更强模型，理解更深 |

#### 实现架构

```
┌─────────────────────────────────────────────────────────────────┐
│                  More Intelligence 实现架构                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【Step 1】反馈收集（数据层）                                    │
│  └── 记录用户的每次反馈：评价、原因、具体信息反馈                 │
│       ↓                                                         │
│  【Step 2】AI 分析反馈 ⭐                                        │
│  └── 定期让 AI 分析用户反馈历史                                  │
│  └── AI 输出：用户喜欢什么、不喜欢什么、偏好变化                 │
│       ↓                                                         │
│  【Step 3】AI 生成用户画像（自然语言）⭐                         │
│  └── AI 用自然语言描述对用户的理解                               │
│  └── 不是结构化标签，是"这个用户是什么样的人"                    │
│       ↓                                                         │
│  【Step 4】Prompt 动态注入                                       │
│  └── 每次生成简报时，把 AI 的理解注入 Prompt                     │
│  └── AI 基于理解自己决定如何筛选                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### Step 1：反馈数据收集

每次用户反馈，记录结构化数据：

```json
{
  "user_id": "user_123",
  "date": "2026-01-11",
  "overall_rating": "negative",
  "reason_selected": ["内容不感兴趣", "漏掉重要信息"],
  "reason_text": "今天没看到 Arbitrum 的空投消息",
  "item_feedbacks": [
    {"item_id": "001", "source": "@whale_alert", "rating": "dislike"},
    {"item_id": "005", "source": "@VitalikButerin", "rating": "like"}
  ]
}
```

#### Step 2：AI 分析反馈（核心）

**Prompt 设计**：让 AI 分析反馈，而不是人写规则

```
你是一个用户偏好分析专家。请分析以下用户的反馈历史。

## 用户当前画像
{current_profile}

## 最近 7 天的反馈记录
{recent_feedbacks}

## 请分析并输出：

1. 用户表达了什么不满？（不喜欢的内容类型、来源）
2. 用户表达了什么喜好？（喜欢的内容类型、来源）
3. 用户对信息量的偏好（多/少）
4. 偏好变化总结（自然语言）
5. 下次简报建议

请用 JSON 格式输出。
```

**AI 分析输出示例**：

```json
{
  "dislikes": {
    "topics": ["Meme 币", "价格预测"],
    "sources": ["@xxx_meme"],
    "reason": "用户多次标记不感兴趣"
  },
  "likes": {
    "topics": ["L2 技术", "空投信息", "融资"],
    "sources": ["@VitalikButerin", "@arbitrum"],
    "reason": "用户主动反馈想看空投，标记技术内容有用"
  },
  "volume_preference": "精简，10-15条",
  "preference_summary": "技术导向的 L2 深度用户，特别关注 Arbitrum。不喜欢 Meme 和价格分析。上次漏掉空投信息导致不满。",
  "suggestions": ["提高 Arbitrum 权重", "增加空投信息关注", "减少 Meme 内容"]
}
```

#### Step 3：AI 生成用户画像（自然语言）

**关键点**：用自然语言描述，而非结构化权重

```yaml
user_profile:
  user_id: "user_123"
  
  # 初始设置
  initial_interests: ["DeFi", "以太坊", "Layer2"]
  
  # ⭐ AI 生成的偏好理解（自然语言描述）
  ai_understanding: |
    这是一个技术导向的 DeFi/L2 深度用户。
    
    **特别关注**：
    - Arbitrum 生态动态，尤其是空投和激励活动
    - 以太坊技术路线图和升级进展
    - L2 赛道的融资和新项目
    
    **明确不喜欢**：
    - Meme 币相关内容（多次标记不感兴趣）
    - 纯价格预测和技术分析
    - 信息量过多，偏好精简
    
    **需要注意**：
    - 上次漏掉 Arbitrum 空投信息，用户不满
    - 应提高空投/激励类信息的覆盖度
    
  last_updated: "2026-01-11"
```

#### Step 4：Prompt 动态注入

每次生成简报时，把 AI 的理解注入：

```
你正在为用户生成今日 Web3 简报。

## 用户基本信息
- 关注领域：DeFi, 以太坊, Layer2
- 关注项目：ETH, ARB, OP

## ⭐ AI 对该用户的深度理解（请仔细阅读并遵循）

这是一个技术导向的 DeFi/L2 深度用户。

**特别关注**：
- Arbitrum 生态动态，尤其是空投和激励活动
- 以太坊技术路线图和升级进展

**明确不喜欢**：
- Meme 币相关内容
- 纯价格预测
- 信息量过多

**需要注意**：
- 上次漏掉 Arbitrum 空投信息，用户不满

## 今日待筛选信息
{today_raw_news}

## 任务
基于对用户的理解，筛选出最有价值的 10-15 条信息。
特别注意不要漏掉 Arbitrum 相关信息，过滤 Meme 内容。
```

#### 为什么这样设计是 "More Intelligence"？

| 维度 | 说明 |
|------|------|
| **不写规则** | 没有 if-else 硬编码，AI 自己理解 |
| **自然语言** | 偏好用人话描述，AI 更容易理解和应用 |
| **AI 决策** | AI 自己决定如何使用偏好，不被规则限制 |
| **可进化** | 换更强的模型，理解能力自动提升 |
| **灵活性** | AI 能理解复杂、模糊的偏好表达 |

#### 实现复杂度

| 模块 | 复杂度 | MVP 可行性 |
|------|--------|------------|
| 反馈收集 | 🟢 简单 | ✅ Telegram Bot 记录点击 |
| 反馈存储 | 🟢 简单 | ✅ 数据库存 JSON |
| AI 分析反馈 | 🟡 中等 | ✅ 定期调用 LLM |
| 画像存储 | 🟢 简单 | ✅ 存文本 |
| Prompt 注入 | 🟢 简单 | ✅ 字符串拼接 |

**结论**：MVP 完全可实现，核心就是多调用几次 LLM。

### 5.3 AI 智能筛选

#### 设计理念

**不是**写一堆规则告诉 AI 怎么筛选。

**而是**：
- 告诉 AI 这个用户关心什么
- 告诉 AI 这条信息是什么
- 让 AI 判断：这条信息对这个用户有价值吗？

#### 筛选逻辑（概念）

```
原始信息池（每天 500+ 条）
        ↓
    AI 理解内容
        ↓
    AI 理解用户
        ↓
    AI 判断价值（打分）
        ↓
    排序 + 去重
        ↓
    精选 15-30 条
```

#### AI 判断依据

| 因素 | 权重 |
|------|------|
| 与用户关注领域的相关度 | 高 |
| 信息的重要程度（行业大事） | 高 |
| 信息源的权威性 | 中 |
| 时效性 | 中 |
| 内容质量（非水文） | 中 |
